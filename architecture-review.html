Architecture Review

Piano BLE Bridge: Renaming was just the start. Let's deepen the modules.

1. Deepen the UI Module

Strong Recommendation

Files: bridge-s3.ino, BridgeUi.cpp, BridgeUi.h

The Problem (Shallow & Leaky)

UI rendering is "orphaned": the .ino file draws the main dashboard because BridgeUi is too shallow to handle it. BridgeUi leaks internal state (counters, logs) to the caller just so it can be drawn.

The Solution (Deepening)

Move all rendering logic into BridgeUi. Provide a deep interface: refresh() for the frame loop and notify(Event) for reactive updates. The .ino file becomes a pure controller.

graph TD subgraph Before: Shallow & Leaky INO[bridge-s3.ino] -->|Read State| UI[BridgeUi] INO -->|Draw Dashboard| GFX[Arduino_GFX] UI -->|Draw Overlays| GFX end subgraph After: Deep UI CONTROLLER[bridge-s3.ino] -->|Notify Event| DEEPUI[BridgeUi] CONTROLLER -->|Refresh| DEEPUI DEEPUI -->|Draw Everything| GFX end

Benefits: Locality of UI logic. bridge-s3.ino is reduced to coordination. Testing UI transitions becomes possible by mocking the GFX interface.

2. Unify Connectivity Manager

Worth Exploring

Files: NetworkServices.cpp, RTPMidiConnection.cpp, WifiProvisioning.cpp

The Problem (Excessive Indirection)

Understanding "Is the network ready?" requires bouncing through three files. NetworkServices is a shallow facade that adds indirection without providing leverage.

The Solution (Locality)

Merge these into a ConnectivityManager. Hide the state machine of WiFi -> Provisioning -> RTP-MIDI. Callers only care about isConnected() and send(MidiPacket).

graph LR subgraph Before: Layered Indirection NS[NetworkServices] --> RTP[RTPMidiConnection] RTP --> WP[WifiProvisioning] end subgraph After: Deep Connectivity CM[ConnectivityManager] subgraph Internal WP2[WiFi/Portal] RTP2[RTP Stack] end CM -.-> WP2 CM -.-> RTP2 end

Benefits: Simplifies the main setup loop. Locality for network state transitions. Easier to add new transports (e.g. Ethernet) without changing the coordinator.

3. MIDI Message Abstraction

Worth Exploring

Files: MidiCodec.h, BridgeSettings.cpp, USBConnection.cpp

The Problem (Scattered Logic)

MIDI transformation (transpose, filter) is coupled with BridgeSettings (persistence). Parsing logic is repeated or scattered. No single "Seam" exists for MIDI processing.

The Solution (Deepening)

Create a MidiMessage module. It handles its own serialization/deserialization. Pair it with a MidiProcessor that handles transformations independently of settings.

Benefits: Transformation logic becomes unit-testable in isolation. BridgeSettings returns to being a pure data layer. Transports only deal with MidiMessage objects.

Top Recommendation: Deepen the UI

The UI fragmentation is the largest source of friction. Moving all rendering into BridgeUi will immediately clarify the role of the .ino file and allow BridgeUi to earn its keep as a deep module. It also enables the introduction of a more robust state machine for UI modes.
